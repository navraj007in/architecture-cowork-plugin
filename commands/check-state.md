# /architect:check-state

Validates `_state.json` for schema compliance, type errors, and per-command write rule violations. Identifies stale fields, missing required sections, and inconsistencies across state.

## Trigger

```
/architect:check-state
/architect:check-state [--fix]     # auto-fix simple issues
/architect:check-state [--strict]  # fail on warnings (not just errors)
```

## Purpose

The `_state.json` file accumulates output from 50+ commands, each owning specific fields. Over time, inconsistencies emerge:
- Field values that don't match their declared type (e.g., `project.stage` is array instead of enum)
- Stale fields (set by command A, never updated, contradicts current command B output)
- Missing required fields for the project's stage
- Violations of per-command write rules (command X overwrote field owned by command Y)

This command scans the state file and reports all issues with severity levels.

## Input

No required input. Reads `architecture-output/_state.json` in the current project.

## Output Files

**1. `architecture-output/state-check-report.md`** (always generated)

```markdown

**Project:** my-startup  
**Stage:** growth

❌ **Critical Errors:** 2
⚠️  **Warnings:** 3

## Critical Issues

### 1. Invalid type: `project.stage` (CRITICAL)
**Field:** `project.stage`  
**Expected:** `"concept" | "mvp" | "growth" | "enterprise"`  
**Actual:** `["mvp", "growth"]` (array, not string!)  
**Consequence:** Downstream commands interpret stage as growth only; this breaks cost estimation and roadmap generation.  
**Fix:** `jq '.project.stage = "growth"' _state.json > _state.json.fixed && mv _state.json.fixed _state.json`

### 2. Stale field: `blueprint.deepen_passes` (CRITICAL)
**Field:** `blueprint.deepen_passes`  
**Value:** `0` (set 30 days ago)  
**Problem:** Current blueprint shows `deepen_passes: 3`, but state shows `0`. This contradicts the actual output.  
**Consequence:** Next blueprint run may not deepen correctly.  
**Fix:** Update to `3` or delete (will be regenerated by next blueprint run).  
**Command:** `jq '.blueprint.deepen_passes = 3' _state.json > _state.json.fixed && mv _state.json.fixed _state.json`

**Field:** `design`  
**Owned by:** `design-system` command  
**Last modified:** 2026-04-20 (blueprint ran on 2026-04-20, may have overwritten design tokens)  
**Consequence:** Design is out of sync with actual design-system output.  
**Suggested fix:** Run `/architect:design-system` to refresh design tokens.

### 2. Missing recommended field: `entities` (WARNING)
**Stage:** growth  
**Recommendation:** growth-stage projects should have `entities` defined (currently absent).  
**Consequence:** Data-model generation will fall back to SDL entities only (may miss business logic).  
**Suggested fix:** Run `/architect:generate-data-model` to populate entities.

### 3. Unused field: `cost_estimate.old_currency_format` (WARNING)
**Field:** `cost_estimate.old_currency_format`

## Behavior

### Step 1: Load and parse
- Read `architecture-output/_state.json`
- Validate JSON syntax
- Stop immediately if invalid JSON (cannot proceed)

### Step 2: Type validation
For each field in the canonical schema (per CLAUDE.md):
- ✅ Field exists and has correct type (string, number, object, array)
- ✅ Enum fields (like `project.stage`) are one of the allowed values
- ❌ Type mismatch → report as CRITICAL

### Step 3: Stage-based requirements
Get `project.stage` and check stage-specific mandatory fields:
- **concept:** requires `project`
- **mvp:** requires `project`, `tech_stack`, `components`, `design`
- **growth:** requires `project`, `tech_stack`, `components`, `entities`, `design`, `blueprint`
- **enterprise:** requires all of the above plus `monitoring`, `compliance`

Missing required field → report as CRITICAL.

### Step 4: Write-rule validation
For each top-level command-owned field (per CLAUDE.md §230):
- Check `last_modified_timestamp` (if available in state)
- Cross-reference with `_activity.jsonl` to see which command last wrote it
- If command A owns field X, but command B's activity shows it also wrote field X → report VIOLATION as CRITICAL

### Step 5: Data validation
- Color fields: must be valid hex (#RRGGBB)
- Numeric fields: must be numbers, not strings
- Enum fields: must be exact match (no substring/case-insensitive)
- Arrays: must be arrays, not strings

### Step 6: Stale field detection
For fields older than 30 days:
- Flag for manual review
- Suggest running the owning command to refresh
- Report as WARNING (not critical, unless causes type conflict)

### Step 7: Generate report
Write comprehensive report with:
- Summary stats (pass rate, critical/warning counts)
- Each issue with detailed context and fix steps
- Recommendations sorted by priority
- Schema validation summary (what passed)

### Step 8: Update activity log
Log this check to `_activity.jsonl`:
```json
{"ts":"<ISO>","phase":"check-state","outcome":"success|warning|failed","critical":N,"warnings":M,"summary":"..."}
```

## Flags

### `--fix`
Auto-fix safe issues:
- ✅ Correct type mismatches (e.g., array → string)
- ✅ Remove unknown fields (not in canonical schema)
- ✅ Fix color format (add missing # prefix if just "f97316")
- ❌ Do NOT auto-fix critical write-rule violations (requires manual investigation)

Before fix:
- Create backup: `_state.json.backup.<timestamp>`
- Run validation script on new state
- If validation fails, revert from backup

After fix:
- Report what was fixed
- Still generate check-state-report.md showing what was done
- Activity log: `outcome: "success"` (issues fixed)

### `--strict`
Fail (non-zero exit) on warnings, not just critical errors.
- Useful for CI/CD gates: "no PR merge if state has warnings"
- Default: exit 0 on warnings, exit 1 on errors

## Example: Full Flow

1. User has made many scaffold and implement runs over 3 weeks
2. User runs `/architect:check-state`
3. Command generates report identifying:
   - Stale `cost_estimate` (30 days old, should refresh)
   - Design colors not applied (design-system hasn't run recently)
   - One type mismatch (component.port is string "3000", should be number)
4. Report suggests: run `/architect:design-system`, run `check-state --fix`, then verify
5. User runs `/architect:check-state --fix`
6. Command fixes type mismatch, generates backup, validates
7. Activity log records: "check-state completed: 1 issue fixed, 2 warnings"
8. User follows recommendations to refresh design tokens
9. Next blueprint run uses consistent, validated state

- Does NOT modify SDL or codebase, only _state.json (with backup)
